Architecture Vocabulary

Module — a unit with an interface and implementation · Depth — interface simplicity vs implementation complexity · Seam — a place where modules meet · Locality — related code living together · Leverage — a small interface change propagating widely · Adapter — a module that translates between two interfaces

1. Deepen src/logic.ts — Split the God Module

Strong

Files: src/logic.ts (630 lines, 30+ exports, 7 sections) · tests/unit/logic.test.ts (450 lines)

Problem

logic.ts is a shallow module masquerading as a deep one. Its interface is enormous — 30+ exports spanning model definitions, API key resolution, WorkOS credential parsing, dynamic model discovery, error classification, and environment helpers. Understanding any single concept requires scrolling past unrelated sections. The locality is poor: the classifyClinePassError function and its message constants live far from the message_end handler in index.ts that calls them. The deletion test passes with force: deleting logic.ts would concentrate all complexity into one place, but that place is too big to navigate.

Solution

Split logic.ts into 4 deep modules along its existing section seams:

  • src/models.ts — Model definitions + dynamic discovery (MODELS, fetchRemoteModels, resolveModels, ModelConfig)
  • src/auth.ts — Credential resolution (resolveApiKey, resolveClineAuthCredentials, isWorkosToken, auth path helpers)
  • src/errors.ts — Error classification (classifyClinePassError, CLINEPASS_ERROR_MESSAGES, ClinePassErrorType)
  • src/env.ts — Environment helpers (resolveApiBase, sanitizeApiKey, buildEndpointUrl, shared type guards)

logic.ts becomes a barrel re-export for backward compat, or is removed if callers import directly.

Benefits

Locality: error classification lives next to the error handler that calls it. Model definitions live next to the discovery logic that fetches them. Test surface: each module has a focused test file instead of one 450-line monster. Leverage: adding a new error type or auth source touches one module, not a 630-line file. AI-navigability: agents can load a single small file to understand a concept.

2. Deepen the Error Surface — Extract a ClinePassErrorHandler

Worth exploring

Files: src/index.ts (error handler block, ~25 lines) · src/logic.ts (classifyClinePassError, ~50 lines)

Problem

The error handling seam is split across two modules with no owning interface. classifyClinePassError in logic.ts maps error strings to types + messages. The message_end handler in index.ts does the provider filtering (msg.provider ?? ctx.model?.provider), the UI/non-UI dispatch (ctx.ui.notify vs console.error), and the classification call. The logic is shallow in both places — the interface of each piece is nearly as complex as its implementation. There's no locality: if you need to understand "what happens when ClinePass returns 403", you read two files. The deletion test barely passes: deleting the handler block from index.ts would move complexity, not concentrate it — the real question is where it should live.

Solution

Create a deep ClinePassErrorHandler module that owns the full error surface: classification, provider filtering, and message delivery. The interface is a single function: 

// src/errors.ts
export function handleClinePassError(
  message: AgentMessage,
  ctx: ExtensionContext,
): void  // returns true if handled, false if not ours

index.ts shrinks to: pi.on("message_end", (e, ctx) => handleClinePassError(e.message, ctx))

Benefits

Locality: classification + filtering + delivery live together. Test surface: the interface is the test — pass a fake message + context, assert the notify call. No need to test the classification and the handler separately. Leverage: adding 403-specific re-login prompts or 429 backoff hints touches one module.

3. Deepen src/oauth.ts — Extract a WorkOS Protocol Adapter

Worth exploring

Files: src/oauth.ts (200 lines) · src/logic.ts (WorkOS constants + resolveClineAuthCredentials)

Problem

oauth.ts mixes two concerns at the same depth: (1) the WorkOS HTTP protocol — the non-standard granttype field, the workos: prefix enforcement, the {data: {...}} vs flat response parsing; and (2) the pi login orchestration — detecting Cline CLI creds, deciding refresh vs paste, wiring callbacks. The WorkOS protocol details are an adapter that should sit behind a seam, but they're inlined in refreshWorkosToken(). Meanwhile, the WorkOS constants (WORKOS_TOKEN_PREFIX, WORKOS_TOKEN_LIFETIME_MS, etc.) live in logic.ts, breaking locality. The deletion test passes: deleting refreshWorkosToken would force the protocol details into a named adapter, concentrating them where they belong.

Solution

Extract a src/workos.ts adapter that owns the entire WorkOS protocol surface:

  • WorkOS constants (prefix, lifetime, refresh margin, refresh endpoint)
  • refreshWorkosToken() — the HTTP call with non-standard format handling
  • isWorkosToken() — prefix detection
  • resolveClineAuthCredentials() — WorkOS credential extraction from Cline CLI config

oauth.ts becomes a pure orchestration layer: "try WorkOS, else paste" — with no protocol knowledge.

Benefits

Locality: all WorkOS protocol knowledge in one place. Adapter depth: the adapter's interface is refresh(token) → token — simple, hiding the non-standard API format. Test surface: WorkOS tests target the adapter directly; OAuth tests test orchestration with a mocked adapter. Leverage: if Cline changes the refresh format (e.g., normalizes granttype → grant_type), one file changes.

4. Deepen Test Locality — Align Test Files with Source Modules

Speculative

Files: tests/unit/logic.test.ts (450 lines, 50+ tests) · tests/unit/index.test.ts (340 lines, 12 tests)

Problem

logic.test.ts tests 7 distinct concept areas (models, auth, WorkOS, discovery, errors, sanitization, constants) in one file. When a model test fails, you scroll past 200 lines of auth tests to find it. The test locality is the inverse of good source locality — everything is in one place, which means nothing is in the right place. index.test.ts mixes provider registration tests with error handler tests. This is a symptom of Candidate 1 (God Module) — the test file mirrors the source file's lack of seams.

Solution

Split test files to mirror the source module split from Candidate 1:

  • tests/unit/models.test.ts — MODELS, modelIds, fetchRemoteModels, resolveModels
  • tests/unit/auth.test.ts — resolveApiKey, resolveClineAuthCredentials, isWorkosToken
  • tests/unit/errors.test.ts — classifyClinePassError, CLINEPASS_ERROR_MESSAGES
  • tests/unit/env.test.ts — resolveApiBase, sanitizeApiKey, buildEndpointUrl
  • tests/unit/registration.test.ts — provider registration shape, oauth wiring
  • tests/unit/error-handler.test.ts — message_end handler tests

Benefits

Locality: when classifyClinePassError breaks, you open errors.test.ts — not a 450-line file. Test surface as interface: each test file is the interface contract for its module. Parallel dev: two developers can work on auth tests and model tests without merge conflicts. Depends on Candidate 1: this is only valuable if the source modules are split first.

🏆 Top Recommendation

Start with Candidate 1 — Split logic.ts. It's the root cause: the God module forces poor locality in both source and tests, makes the error surface split (Candidate 2), and scatters WorkOS constants (Candidate 3). Splitting it unblocks all other candidates naturally — the error handler, the WorkOS adapter, and the test files all fall out of the module boundaries you draw.

The split is low-risk: logic.ts can remain as a barrel re-export for backward compatibility, so index.ts and oauth.ts imports don't change. The ADRs (especially ADR-0002 on pure logic/IoC) are fully respected — the DI pattern moves with each function into its new module.
